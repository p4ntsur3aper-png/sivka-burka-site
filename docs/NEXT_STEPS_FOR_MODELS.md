# Следующие шаги для моделей

## Рекомендуемый первый пакет работ

Начинать лучше не с новых экранов, а с укрепления редактирования контента. Это главный разрыв между текущим состоянием и ожиданием "админ может менять все на сайте".

### Пакет 1. CMS v1

Задачи:

- TASK-0101: единая модель контента;
- TASK-0102: content repository;
- TASK-0103: MediaPicker;
- TASK-0104: редактирование всех блоков главной.

Почему сначала это:

- Сейчас данные разложены по `SiteContent`, mock-массивам и JSX.
- Без единой модели каждая новая правка будет точечной.
- Для backend-разработчика нужен понятный контракт.

Ожидаемый результат:

- админ может редактировать главную страницу в контексте публичной страницы;
- изображения выбираются через загрузку/медиатеку;
- блоки можно скрывать и менять местами;
- компоненты читают данные через repository.

## Рекомендуемый второй пакет работ

### Пакет 2. Управляемые разделы сайта

Задачи:

- TASK-0105: контакты, отзывы, правила и FAQ;
- TASK-0201: альбомы галереи;
- TASK-0202: пакетная загрузка фото.

Ожидаемый результат:

- галерея выглядит как набор альбомов;
- админ может управлять всеми публичными справочными разделами;
- нижние папки галереи становятся осмысленной навигацией, а не декоративным блоком.

## Рекомендуемый третий пакет работ

### Пакет 3. Правила записи без JSON

Задачи:

- TASK-0301: формы для правил записи;
- TASK-0302: решение по инструкторам и другим ресурсам.

Ожидаемый результат:

- админ может закрыть день, задать перерыв, поменять рабочие часы и недоступность лошади через понятные поля;
- mock-логика доступности остается в `availabilityService.ts`;
- frontend готов к серверной проверке.

## Рекомендуемые промпты для отдельных моделей

### Для архитектора

```text
Изучи docs/BUSINESS_AUDIT.md, docs/CONTENT_EDITING_DEEP_DIVE.md и текущие src/types, src/services.
Сделай TASK-0101 и TASK-0102. Не меняй визуальный дизайн. Цель - единая модель контента и repository-слой, чтобы потом подключить backend.
После работы запусти npm run build и кратко опиши изменения.
```

### Для UI/UX модели

```text
Изучи текущую админку и docs/CONTENT_EDITING_DEEP_DIVE.md.
Предложи UX для режима редактирования на публичных страницах: боковая панель, кнопки редактирования, медиатека, сортировка блоков.
Не реализуй код, подготовь docs/ADMIN_EDITING_UX.md с экранами, состояниями и acceptance criteria.
```

### Для frontend-реализатора

```text
Реализуй TASK-0103 MediaPicker.
Нужно заменить сценарий "ввести путь к файлу" на выбор из медиатеки и загрузку файла.
Сохрани mock-режим без backend. Проверь фото главной, услуги, лошади и галереи.
```

### Для QA модели

```text
Изучи docs/TASKS.md и текущий проект.
Создай docs/QA_CHECKLIST.md. Проверь сборку, основные маршруты, клиентскую запись, админ-редактирование фото и управление заявкой.
Пиши баги в формате: шаги, ожидаемо, фактически, приоритет.
```

## Что не делать в ближайших итерациях

- Не делать полноценный backend.
- Не добавлять сложную авторизацию, если задача только про frontend.
- Не переписывать весь проект с нуля.
- Не переносить бизнес-логику доступности в React-компоненты.
- Не хранить реальные персональные данные в `localStorage`.
- Не добавлять тяжелые библиотеки без явной необходимости.
