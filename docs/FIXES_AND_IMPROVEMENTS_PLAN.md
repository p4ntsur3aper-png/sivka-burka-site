## TASK-0701. Исправить верстку вкладки "Контент" в админке

Приоритет: P0  
Роль: Frontend UI Engineer  
Файлы: `src/pages/AdminPage.tsx`, `src/styles/global.css`

### Проблема

Во вкладке "Контент" при редактировании через админку элементы в верхней части страницы расплываются и наслаиваются друг на друга. По описанию проблема проявляется в первом большом блоке до секции отзывов.

Вероятные причины:

- слишком плотная сетка для форм;
- карточки/поля не имеют устойчивых `min-width`, `grid-template-columns` или `flex-wrap`;
- изображения и поля редактирования оказываются в одной строке без нормальных ограничений;
- длинные textarea/input растягивают контейнер;
- мобильные и desktop-стили пересекаются.

### Что сделать

- Проверить вкладку "Контент" на ширинах 390, 768, 1280 px.
- Найти блоки, где элементы накладываются друг на друга.
- Разделить крупные формы на понятные секции: текст, медиа, действия.
- Для редакторских карточек задать устойчивую сетку:
  - одна колонка на мобильных;
  - две колонки только там, где хватает места;
  - `minmax(0, 1fr)` для колонок;
  - `overflow-wrap: anywhere` для длинных значений;
  - фиксированные ограничения для preview-изображений.
- Убедиться, что кнопки сохранения, загрузки фото и поля ввода не перекрывают друг друга.

### Критерии готовности

- Вкладка "Контент" читаема на мобильном, планшете и desktop.
- До блока "Отзывы" нет наложений элементов.
- Изображения не раздувают форму.
- Поля ввода не выходят за границы карточек.
- `npm run build` проходит.

## TASK-0702. Улучшить UX клиентской записи и календаря

Приоритет: P0  
Роль: UX + Frontend Engineer  
Файлы: `src/components/booking/BookingForm.tsx`, `src/services/availabilityService.ts`, `src/styles/global.css`

### Проблема

Сейчас клиентская запись воспринимается немного неудобно. Дату лучше выбирать через привычный календарный интерфейс, а не как обычное поле или разрозненный набор элементов.

Целевое поведение:

1. Клиент выбирает услугу.
2. Нажимает на поле даты.
3. Открывается календарь.
4. Недоступные даты визуально отличаются серым цветом.
5. Доступные даты активны.
6. После выбора даты ниже показываются доступные временные слоты.
7. Недоступные слоты можно показать неактивными с причиной.
8. После выбора времени пользователь заполняет участников и контактные данные.

### Важное уточнение

Стандартный браузерный `input type="date"` нельзя полноценно стилизовать так, чтобы закрашивать отдельные недоступные даты серым во всех браузерах. Поэтому для нужного UX лучше сделать небольшой собственный календарь или подключить легкую библиотеку.

Для дипломного проекта оптимальный вариант: собственный простой календарь на месяц без тяжелой зависимости.

### Что сделать

- Добавить компонент `BookingCalendar`.
- Показывать текущий месяц и переключение месяцев.
- Для каждой даты вычислять статус:
  - доступна;
  - недоступна;
  - выбрана;
  - прошедшая дата;
  - закрытый день.
- Серым показывать недоступные и прошедшие даты.
- При наведении/фокусе показывать короткую причину недоступности, если она есть.
- После выбора даты загружать/пересчитывать слоты через `availabilityService`.
- Слоты показывать отдельным блоком: утро, день, вечер или простой список кнопок.
- Если доступного времени нет, показывать понятное сообщение без технических деталей.
- Оставить текст, что заявка является предварительной и ожидает подтверждения администратора.

### Что не делать на этом шаге

- Не делать автоматическое финальное бронирование.
- Не переносить бизнес-логику доступности в React-компонент.
- Не усложнять календарь drag-and-drop возможностями.

### Критерии готовности

- Дата выбирается через календарный интерфейс.
- Недоступные даты визуально серые и не выбираются.
- После выбора даты показываются слоты времени.
- Для недоступных слотов есть понятные причины.
- Несколько участников продолжают работать.
- Проверка веса, тренеров и лошадей не ломается.
- `npm run build` проходит.

## TASK-0703. Переработать вход для администратора и тренера

Приоритет: P0  
Роль: Frontend UX + Auth Mock Engineer  
Файлы: `src/components/layout/Layout.tsx`, `src/pages/TrainerLoginPage.tsx`, `src/pages/AdminPage.tsx`, `src/services/trainerAuth.ts`, возможно `src/services/adminAuth.ts`

### Проблема

Сейчас вход выглядит как кнопка "Админ", хотя в системе уже есть не только администратор, но и тренер. Для публичного сайта такая заметная кнопка авторизации выглядит неестественно: обычному клиенту она не нужна.

### Предлагаемая логика

Сделать один служебный маршрут входа:

- `/login` или `/staff/login`

На странице входа пользователь выбирает роль:

- Администратор;
- Тренер.

Для mock-версии:

- администратор входит по демонстрационной кнопке или простому паролю;
- тренер выбирает себя из списка тренеров, как сейчас, но внутри общего входа.

После входа:

- администратор попадает в `/admin`;
- тренер попадает в `/trainer/schedule`.

### Что сделать в публичном интерфейсе

- Убрать заметную кнопку "Админ" из основной навигации.
- Добавить менее навязчивую ссылку для сотрудников:
  - в футере: "Вход для сотрудников";
  - либо маленькой ссылкой в нижней части меню.
- Не показывать клиенту административную логику как часть основного пользовательского сценария.

### Что сделать технически

- Добавить общий mock-сервис сессии или аккуратно расширить текущий `trainerAuth`.
- Разделить понятия:
  - публичный клиент без входа;
  - администратор;
  - тренер.
- Добавить route guard на уровне страниц:
  - если нет admin-сессии, `/admin` просит войти;
  - если нет trainer-сессии, trainer-страницы просят войти.
- В демо-режиме не усложнять настоящей безопасностью, но честно оставить комментарий/документацию, что реальная авторизация будет на backend.

### Критерии готовности

- В публичной шапке нет крупной кнопки "Админ".
- Есть единая понятная страница входа для сотрудников.
- Админ и тренер входят разными сценариями.
- Тренер не попадает в админку.
- Администратор не выглядит как единственная роль системы.
- `npm run build` проходит.

## TASK-0704. Добавить поддержку видео-контента

Приоритет: P1  
Роль: Frontend Media Engineer  
Файлы: `src/types/index.ts`, `src/services/contentRepository.ts`, `src/components/admin/MediaPicker.tsx`, `src/pages/HomePage.tsx`, `src/pages/GalleryPage.tsx`, `src/styles/global.css`

### Ответ на вопрос

Да, видео-содержимое подключить возможно. Для текущего проекта лучше поддержать два варианта:

- загрузка локального видеофайла в mock-медиатеку;
- вставка внешней ссылки, например VK Video, RuTube или другой embed/url.

Для дипломной версии безопаснее начать с внешних ссылок и небольших локальных файлов, потому что полноценное хранение больших видеофайлов должен делать backend или отдельное файловое хранилище.

### Что добавить в типы

Расширить медиа-модель:

```ts
type MediaAssetType = 'image' | 'video';

interface MediaAsset {
  id: string;
  fileName: string;
  title: string;
  altText?: string;
  url: string;
  type: MediaAssetType;
  mimeType?: string;
  sizeBytes?: number;
  folderId?: string;
  posterUrl?: string;
  durationSeconds?: number;
  createdAt: string;
}
```

Для видео важно добавить:

- `posterUrl` для обложки;
- `durationSeconds`, если известно;
- `type: 'video'`.

### Что сделать в админке

- В `MediaPicker` добавить фильтр: изображения / видео / все.
- Разрешить загрузку файлов `video/mp4`, `video/webm`.
- Для видео показывать preview через `<video controls muted>`.
- Добавить поле обложки видео.
- Разрешить вставку внешней ссылки.
- Для внешних ссылок хранить URL и тип `video`.

### Где использовать видео

Минимальный полезный набор:

- Главная страница: видео-блок "Как проходит занятие".
- Галерея: отдельный раздел "Видео".
- Карточка услуги: опциональное видео услуги.

### Ограничения mock-версии

- Большие видео нельзя надежно хранить в `localStorage`, поэтому локальные файлы лучше хранить как object/data URL только для демо и небольших файлов.
- Для реального backend нужен `multipart/form-data` upload и файловое хранилище.
- Для внешних платформ нужно согласовать формат embed-ссылок.

### Критерии готовности

- В медиа-модели есть различие image/video.
- В админке можно добавить видео или ссылку на видео.
- В публичной части видео отображается аккуратно и адаптивно.
- Если видео недоступно, интерфейс не ломается.
- В API-контракте описаны поля видео.
- `npm run build` проходит.
