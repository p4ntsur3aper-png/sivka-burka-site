# План развития: тренерские учетные записи и уведомления

Документ описывает следующий крупный этап развития проекта после клиентской записи, календаря бронирования, управления лошадьми и CMS-режима. Цель этапа - показать, что web-приложение учитывает не только клиента и администратора, но и ежедневную работу тренеров конюшни.

## Почему это важно

Сейчас заявка проходит путь от клиента к администратору. Для реальной конюшни этого недостаточно: после подтверждения записи нужно понять, кто из тренеров проводит занятие, свободен ли он, получил ли уведомление, готов ли к занятию и видит ли детали участников.

Тренерская роль закрывает несколько бизнес-задач:

- снижает риск забытых или нераспределенных занятий;
- позволяет администратору видеть загрузку сотрудников;
- помогает тренеру заранее узнать состав группы, возраст, вес, уровень подготовки и особые комментарии;
- создает понятный сценарий подтверждения занятия внутри предприятия;
- готовит проект к реальной интеграции с backend и уведомлениями.

## Роли пользователей

### Клиент

Публичный пользователь сайта. Может смотреть услуги, правила, галерею, контакты и отправлять заявку на запись.

### Администратор

Управляет контентом сайта, услугами, лошадьми, правилами записи, заявками и расписанием. Назначает лошадей и тренеров.

### Тренер

Видит только свои занятия и связанные с ними детали. Может подтвердить, что ознакомился с записью, запросить уточнение или отметить занятие как выполненное.

## Основные сценарии

### 1. Назначение тренера на заявку

1. Клиент отправляет заявку.
2. Администратор открывает календарь бронирований.
3. Администратор проверяет доступность лошадей и тренеров.
4. Администратор назначает тренера на заявку.
5. Тренер получает уведомление в своем кабинете.
6. Тренер открывает детали занятия и подтверждает ознакомление.

### 2. Изменение времени занятия

1. Администратор меняет дату или время заявки.
2. Система повторно проверяет доступность лошадей и тренера.
3. Тренер получает уведомление об изменении.
4. В карточке занятия отображается история изменения времени.

### 3. Уточнение по заявке

1. Тренер видит, что для занятия не хватает информации.
2. Тренер нажимает "Требуется уточнение".
3. Администратор получает уведомление.
4. Статус заявки меняется на `needs_clarification`.

### 4. Напоминание перед занятием

1. За 24 часа или за 2 часа до занятия создается уведомление.
2. Тренер видит напоминание в кабинете.
3. В будущем backend сможет отправлять такое уведомление в Telegram, WhatsApp, SMS или email.

## Страницы и разделы интерфейса

### Для администратора

- вкладка "Тренеры";
- список тренеров;
- добавление и редактирование тренера;
- статус тренера: активен, временно недоступен, отпуск, больничный;
- услуги, которые может проводить тренер;
- рабочие дни и часы;
- заметка администратора;
- назначение тренера в календаре бронирований;
- журнал уведомлений.

### Для тренера

Рекомендуемые маршруты:

- `/trainer/login` - демонстрационный вход тренера;
- `/trainer/schedule` - расписание тренера;
- `/trainer/bookings/:id` - детали занятия;
- `/trainer/notifications` - уведомления;
- `/trainer/profile` - профиль и контактные данные.

Для дипломной версии можно сделать простой демо-вход без полноценной авторизации, но архитектурно отделить его от администратора.

## Mock-логика на frontend

На frontend-этапе не нужно делать реальную отправку SMS, WhatsApp или Telegram. Достаточно:

- хранить тренеров в mock-данных и `localStorage`;
- хранить уведомления в mock-репозитории;
- показывать счетчик непрочитанных уведомлений;
- создавать уведомление при назначении тренера;
- создавать уведомление при изменении времени;
- показывать тренеру только его занятия;
- не хранить реальные персональные данные в постоянном браузерном хранилище в production-логике.

Browser Notification API можно оставить как необязательное улучшение и включать только после явного согласия пользователя.

## Влияние на проверку доступности

Сейчас доступность строится вокруг услуг, времени, лошадей, веса, перерывов и правил. После добавления тренеров нужно учитывать еще один ресурс.

Слот должен быть недоступен, если:

- нет подходящей свободной лошади;
- нет свободного тренера для выбранной услуги;
- выбранный тренер не работает в этот день;
- тренер уже назначен на другое занятие в это время;
- тренер не проводит выбранный тип услуги.

Причины недоступности должны быть понятными:

- "Нет свободного тренера для выбранного времени";
- "Тренер не проводит выбранную услугу";
- "Тренер уже назначен на другое занятие";
- "В этот день тренер недоступен".

## Этапы реализации

### Этап 1. Frontend mock-версия

Добавить типы, mock-данные, сервисы, вкладку тренеров в админке, назначение тренера на заявку и простую страницу расписания тренера.

### Этап 2. Уведомления внутри интерфейса

Добавить mock-уведомления, список уведомлений, счетчик непрочитанных, создание уведомлений при ключевых событиях.

### Этап 3. Проверка доступности с учетом тренеров

Расширить `availabilityService`, чтобы он проверял не только лошадей, но и тренеров.

### Этап 4. Backend-интеграция

Заменить mock-репозитории на реальные API-запросы. На этом этапе тренерские уведомления должны создаваться на сервере, а не только в браузере.

## Рекомендуемый порядок

1. Сначала добавить сущность тренера и mock-репозиторий.
2. Затем добавить вкладку "Тренеры" в админке.
3. Потом добавить назначение тренера в карточке заявки.
4. После этого сделать кабинет тренера.
5. Затем добавить mock-уведомления.
6. В конце расширить проверку доступности.

Такой порядок лучше для дипломного проекта: каждая итерация дает видимый результат, но не требует сразу строить полноценную backend-авторизацию.
