# План задач для дальнейшей разработки

Документ предназначен для работы несколькими моделями или несколькими участниками. Каждая задача должна выполняться отдельной веткой или отдельным коротким циклом: анализ, реализация, проверка, краткий отчет.

## Легенда приоритетов

- P0 - критично для демонстрации и защиты.
- P1 - важно для полноты бизнес-функции.
- P2 - улучшение качества и удобства.

## Этап 0. Стабилизация

Роль: QA/Reviewer

Что сделать:

Критерии готовности:

- Сборка проходит.
- Есть короткий отчет о текущих маршрутах.
- Найдены критичные ошибки интерфейса, если они есть.

Приоритет: P0

## Этап 3. Бронирование и ресурсы

### TASK-0301. Сделать правила записи редактируемыми через формы

Приоритет: P1  
Роль: Frontend Engineer  
Файлы: `src/pages/AdminPage.tsx`, `src/services/availabilityService.ts`

Что сделать:

- Заменить редактирование JSON на формы по типам правил.
- Для рабочего времени: дни недели, начало, конец, шаг слота.
- Для перерыва лошади: минуты отдыха.
- Для закрытого дня: дата и причина.
- Для недоступности лошади: лошадь, дата, интервал, причина.

Критерии готовности:

- Администратор не видит JSON как основной способ редактирования.
- Изменения правил влияют на доступные слоты.

Критерии готовности:

Приоритет: P1  
Роль: Product/QA  
Результат: `docs/DEMO_SCRIPT.md`

- клиент выбирает услугу и отправляет заявку;
