# Аудит бизнес-задач фронтенда

Дата аудита: 31.05.2026  
Проект: клиентская часть web-приложения для конно-спортивных услуг ИП Орлова Н.И.

## Краткий вывод

Текущий фронт уже решает основные демонстрационные задачи дипломного проекта: клиент может изучить услуги, открыть карточку услуги, посмотреть правила, контакты, галерею и оставить предварительную заявку. Самая сильная часть проекта - mock-логика календарного бронирования с учетом лошадей, веса, возраста, статусов, длительности услуги и перерывов между занятиями.

При этом проект пока не закрывает бизнес-задачу управления сайтом полноценно. Админ-режим и inline-редактирование есть, но сейчас это скорее frontend-прототип без настоящей CMS-модели: данные хранятся в `localStorage`, нет медиатеки, черновиков, истории изменений, публикации, ролей, защиты от случайных удалений и редактирования всех текстовых блоков как структурированного контента.

## Проверка бизнес-реалистичности бронирования

Сильные стороны:

- Слот зависит не только от даты и времени, но и от доступности лошадей.
- Учитывается максимальный вес наездника для конкретной лошади.
- Для группы участников требуется несколько подходящих лошадей.
- Учитывается статус лошади: доступна, недоступна, отдых, лечение, занята.
- Есть правило перерыва после предыдущего занятия.
- Есть рабочие часы, закрытые даты и исключения.
- Пользователь видит причины недоступности слота.

Ограничения текущего mock-уровня:

- Нет инструкторов как отдельного ресурса.
- Нет площадок/манежей/маршрутов как ресурсов.
- Нет погодных ограничений и сезонности.
- Нет реального параллельного доступа: два клиента могут выбрать слот в разных браузерах, пока backend не проверит заявку.
- Нет оплаты, предоплаты и отмены со стороны клиента.
- Нет уведомлений администратору и клиенту.
- Нет отдельного статуса "забронировано предварительно на время обработки".

## Проверка админ-редактирования

Сейчас есть два подхода:

1. Отдельная страница `/admin` с вкладками.
2. Inline-режим, где на публичных страницах появляются панели редактирования.

Это хорошая демонстрационная база. Однако для реальной бизнес-функции "администратор может менять сайт без программиста" нужно усилить модель:

- редактируемым должен быть каждый публичный текстовый блок;
- изображения должны загружаться в медиатеку, а не храниться Data URL в браузере;
- у каждой картинки нужны название, alt-текст, категория, папка, дата загрузки;
- нужен режим "черновик - предпросмотр - публикация";
- нужны подтверждения удаления;
- нужна история изменений или хотя бы "отменить последнее изменение";
- нужны отдельные сущности для контактов, отзывов, правил, FAQ, блоков главной страницы;
- нужно убрать ручное редактирование JSON для правил записи и заменить его понятными формами.

## Главные продуктовые риски

| Риск | Влияние | Приоритет |
|---|---:|---:|
| Данные админки живут только в браузере | Изменения пропадут на другом ПК или после очистки браузера | Высокий |
| Нет единой CMS-модели контента | Нельзя гарантировать, что "все можно редактировать" | Высокий |
| Data URL вместо файлового хранилища | Страница и localStorage могут стать тяжелыми, изображения нельзя переиспользовать | Высокий |
| Правила записи редактируются через JSON | Неподходяще для реального администратора | Высокий |
| Контакты, отзывы, правила безопасности не управляются полноценно | Админ-функция выглядит неполной | Средний |
| Нет backend-подтверждения доступности | Mock подходит для диплома, но не для эксплуатации | Высокий |
| Нет тест-кейсов по ключевым сценариям | Трудно контролировать регрессии при дальнейшей работе несколькими моделями | Средний |

## Рекомендованный фокус следующих работ

1. Сначала привести редактирование контента к единой модели: страницы, блоки, поля, изображения, порядок блоков.
2. Затем сделать медиатеку с загрузкой файлов, альбомами, обложками и повторным использованием изображений.
3. После этого заменить сложные JSON-настройки админки на понятные формы.
4. Отдельно подготовить контракт backend API для контента, медиа, бронирований, лошадей, правил и отзывов.
5. В конце добавить тест-матрицу и демо-сценарии для защиты диплома.
