# Работа несколькими моделями

## Зачем нужен этот документ

Проект уже содержит несколько разных направлений: UI, бизнес-логика бронирования, админ-режим, контент, медиатека, API-контракт и дипломная документация. Чтобы несколько моделей не мешали друг другу, работу лучше делить по ролям и коротким задачам.

## Роли моделей

### Model A. Product Analyst

Фокус:

- бизнес-сценарии конюшни;
- полнота требований;
- пользовательские сценарии клиента и администратора;
- acceptance criteria.

Типовые задачи:

- уточнить, какие сценарии реально нужны для ИП Орлова Н.И.;
- проверить, не выглядит ли запись как автоматическое бронирование;
- подготовить демо-сценарий для защиты;
- описать ограничения mock-версии.

### Model B. Frontend Architect

Фокус:

- структура типов;
- разбиение на services/repositories;
- подготовка к backend API;
- снижение дублирования.

Типовые задачи:

- спроектировать `ContentBlock`, `MediaAsset`, `GalleryAlbum`;
- вынести работу с контентом из компонентов;
- подготовить API-контракт;
- проверить, что бизнес-логика не живет в JSX.

### Model C. UI/UX Designer

Фокус:

- админ-режим как редактирование "на месте";
- удобство форм;
- мобильная версия;
- визуальная структура галереи и карточек.

Типовые задачи:

- предложить боковую панель редактирования;
- улучшить медиатеку;
- проверить читаемость и размеры изображений;
- подготовить UX для правил записи без JSON.

### Model D. Frontend Implementer

Фокус:

- реализация компонентов;
- CSS;
- интеграция с mock services;
- сохранение существующего стиля.

Типовые задачи:

- сделать `MediaPicker`;
- подключить редактирование блоков главной;
- добавить управление контактами и отзывами;
- реализовать сортировку элементов.

### Model E. QA Engineer

Фокус:

- регрессии;
- сборка;
- адаптивность;
- проверка сценариев.

Типовые задачи:

- прогнать `npm run build`;
- проверить маршруты;
- описать баги с шагами воспроизведения;
- проверить мобильную форму записи.

## Правила передачи задач

Каждая модель должна оставлять результат в одном из форматов:

```md
## Что проверено

## Что изменено

## Файлы

## Риски

## Что делать дальше
```

Если модель меняет код, она должна указать:

- какие сценарии проверены;
- запускалась ли сборка;
- какие ограничения остались;
- какие файлы нельзя трогать следующей модели без повторного чтения.

## Рекомендуемый порядок работы

1. Product Analyst обновляет `BUSINESS_AUDIT.md` и уточняет приоритеты.
2. Frontend Architect проектирует типы и repository-слой.
3. UI/UX Designer описывает UX админ-режима и медиатеки.
4. Frontend Implementer делает одну группу задач за раз.
5. QA Engineer проверяет маршруты и сценарии.
6. Backend Contract Analyst обновляет API-контракт.

## Как не конфликтовать в файлах

Лучше распределять зоны:

- типы и контракты: `src/types/index.ts`, `docs/API_CONTRACT.md`;
- исходные seed-данные: `backend/src/seed-data.js`;
- бизнес-логика: `src/services/*`;
- публичные страницы: `src/pages/*`;
- админ-компоненты: `src/components/admin/*`;
- стили: `src/styles/global.css`;
- документация: `docs/*`.

Если две модели должны менять один файл, сначала одна модель делает только анализ и пишет план, затем вторая реализует.

## Формат задач для следующей модели

```md
Задача: TASK-0103 MediaPicker
Контекст: сейчас ImageUploadButton возвращает Data URL и не имеет медиатеки.
Нужно: добавить MediaPicker, mock-медиатеку и выбор существующего изображения.
Файлы: src/components/admin/*, src/services/contentRepository.ts, src/types/index.ts.
Проверка: заменить фото главной, услуги, лошади и галереи.
Не делать: backend и реальную авторизацию.
```

## Definition of Done

Задача считается завершенной, если:

- код собирается через `npm run build`;
- пользовательский сценарий проверен вручную;
- данные не захардкожены в компоненте, если это бизнес-данные;
- mock-логика вынесена в `services` или `data`;
- новая функциональность описана в документации или задачах;
- нет ухудшения мобильной версии.
